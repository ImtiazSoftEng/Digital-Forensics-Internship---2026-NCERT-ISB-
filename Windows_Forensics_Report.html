Summary

The main lesson of **Module 1** is that **digital forensics is the science of reconstructing past events using digital evidence**. Your job is not to search for suspicious files or learn forensic tools—it is to answer questions like *what happened, who performed the action, when it happened, how it happened, and what evidence supports that conclusion*. Every action a user performs on a Windows system leaves behind one or more digital traces, called **artifacts**, because Windows records information to operate efficiently, not to help investigators. These artifacts are scattered across different parts of the operating system, and no single artifact tells the complete story. Like a detective investigating a crime or a historian reconstructing past events, a forensic examiner gathers many small pieces of evidence, correlates them, and builds a timeline that explains the sequence of events. Throughout an investigation, you must distinguish **facts** from **assumptions**: a fact is something directly supported by evidence (for example, a program was executed), while an assumption is an interpretation that requires additional supporting evidence (for example, concluding the user stole data). Professional investigators never jump to conclusions or rely on a single artifact; instead, they verify findings using multiple independent sources before reaching a conclusion. Ultimately, the goal of a forensic investigation is not to collect artifacts, but to transform reliable digital evidence into an accurate, objective, and defensible reconstruction of user activity.

The Two Main Types of Evidence


Most Windows evidence falls into two categories.


1. File-Based Artifacts


These are actual files.


Examples:


Prefetch files

Event Log files

Recycle Bin metadata

Browser databases


These exist as files on disk.


2. Registry-Based Artifacts


Windows also stores configuration and history inside the Registry.


Think of the Registry as Windows' central database for settings and certain usage history.


Examples include:


Recently used commands

USB device history

UserAssist

Installed software

User preferences


Later, we'll explore the Registry in depth because it is one of the richest sources of forensic evidence.


Windows Is Always Recording


Even when the user isn't actively working, Windows continues recording events.


Examples include:


System startup

User login

Service starts

Network connections

Device connections

Driver loading

System shutdown


This means a forensic timeline includes both user actions and system actions.


A good investigator learns to distinguish between them.



main points

Windows is not a single program; it is a collection of components working together.


Every user action passes through the operating system before reaching the hardware.


Windows creates records because it needs them for performance, reliability, security, and usability—not for forensic investigations.


A single user action often generates multiple artifacts in different locations.


Evidence is stored in both files (such as Event Logs and Prefetch) and the Registry.


Understanding how Windows processes an action is more valuable than memorizing where an artifact is stored


Summary


Windows is a collection of interconnected components that manage user requests, applications, storage, and hardware. Whenever a user performs an action—such as logging in, opening a document, or connecting a USB device—that action is processed by the operating system, which updates various internal components like the file system, Registry, logs, and caches. These updates create digital artifacts that investigators later analyze. Because each Windows component has its own responsibility, evidence is distributed across many locations rather than stored in one central history file. A forensic investigator's job is to understand how a single action travels through the operating system and to combine the resulting artifacts into an accurate reconstruction of what occurred. This understanding forms the foundation for analyzing specific artifacts in the next module

Artifact 1 — Event Logs


Let's start with the most important one.


What are Event Logs?


Windows constantly records important events.


Imagine a security guard writing everything into a notebook.


09:00


Employee entered building.


09:15


Power failure.


09:30


Visitor arrived.


10:00


Employee left.


Windows does exactly this.


Why does Windows create Event Logs?


Not for investigators.


It creates them so administrators can:


Diagnose problems

Monitor security

Detect failures

Troubleshoot systems


Investigators later use the same logs as evidence.


What can Event Logs tell us?


Many things.


Examples:


User login

User logout

System startup

Shutdown

Failed logins

Password changes

Account creation

USB-related events

Service starts

Software installation


Think of Event Logs as:


The diary of the operating system.


What Event Logs CANNOT tell you


Suppose the logs show:


Word started.


Can you conclude:


The user edited a confidential document.


No.


Event Logs don't tell the whole story.


They only record certain events.


DLLS

shared linked libraries that program used .


Some common DLLs include:


kernel32.dll → Basic operating system services (memory management, file operations, process creation)

user32.dll → Windows, buttons, keyboard, mouse, user interface

gdi32.dll → Graphics drawing

advapi32.dll → Advanced Windows services like registry and security


When you run an application, Windows loads the DLLs it depends on.



Artifact 2 — Prefetch


This is probably the favorite artifact of many Windows investigators.


Why does Prefetch exist?


Windows wants programs to open faster.


So when a program runs,


Windows remembers


which files it used

which DLLs it loaded

how it started


Next time,


Windows loads those resources more efficiently.


That's why programs start faster.


Performance—not forensics—is the purpose.


What does Prefetch tell us?


It answers:


Was this application executed?


Examples:


Chrome


Word


PowerShell


CMD


Explorer


Paint


It can also reveal:


execution count

recent execution times (depending on Windows version)

executable path

What Prefetch cannot prove


Suppose Prefetch contains:


CMD.EXE


It only proves:


CMD executed.


It does not prove:


who used it

what commands were typed

why it executed


Those questions require other artifacts


Artifact 3 — UserAssist


This one confuses many beginners.


Let's make it simple.


Imagine two ways Word can start.


Method 1


User double-clicks Word.


↓


Windows Explorer launches Word.


Method 2


A scheduled task launches Word automatically.


UserAssist mainly records programs that are launched through normal user interaction with the Windows graphical interface (GUI).


That makes it very useful for understanding what a user likely started themselves.


Why does Windows create UserAssist?


To help Windows keep track of user interactions, such as frequently used programs and shortcuts.


Again,


Windows isn't trying to help investigators.


What UserAssist tells us


It often provides:


GUI-launched applications

execution counts

timestamps

information tied to the user profile

What UserAssist cannot prove


Finding Word in UserAssist supports that it was launched through the GUI.


It doesn't tell you:


which document was edited

what changes were made

whether data was copied

Difference Between Prefetch and UserAssist


This is one of the most common interview questions.


Prefetch UserAssist

Performance optimization User activity tracking for the Windows shell

Records program execution Records many GUI-launched programs

System-wide User-specific

Focuses on executable behavior Focuses on user interaction through Explorer


Think of it this way:


Prefetch asks:


"Did this program run?"


UserAssist asks:


"Did this user likely launch this program through the GUI?"

Windows forensic artifacts are the digital footprints left behind by normal operating system activity. Each artifact exists for a specific Windows function—such as improving performance, remembering user preferences, or recording security events—but investigators use them as evidence. No single artifact tells the complete story, and each has limitations. The strength of a forensic investigation comes from correlating multiple artifacts to answer investigative questions, build an accurate timeline, and support conclusions with evidence rather than assumptions
