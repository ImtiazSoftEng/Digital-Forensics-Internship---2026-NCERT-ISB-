Week 3 Summary – File-System Forensics & Data Recovery

This project was not simply about creating files or running Linux commands. Its real objective was to understand how a forensic investigator prepares a realistic evidence dataset that can later be analyzed during a digital forensic investigation.

We began by designing a realistic investigation scenario. Instead of randomly creating files, we imagined a corporate insider threat where an employee named Alex Carter was suspected of attempting to hide confidential company information before leaving the organization. Having a realistic scenario gave purpose to every action we performed throughout the project. Every file, folder, and modification represented user behavior that an investigator might encounter in a real case.

After planning the scenario, we created a raw disk image using the Linux dd command. A disk image is simply a byte-for-byte representation of an entire storage device stored inside a single file. At this stage, the image contained nothing except zero bytes. It had no filesystem, no directories, no files, and no metadata. It was equivalent to purchasing a brand-new hard drive that had never been used.

However, a blank storage device cannot store files. Therefore, the next step was creating an Ext4 filesystem using the mkfs.ext4 utility. This was one of the most important concepts of the project. A filesystem is what transforms raw storage into an organized structure capable of storing files and directories. During this process, Linux automatically created important internal structures such as the superblock, inode tables, block groups, journals, and allocation bitmaps. These structures are extremely important in digital forensics because they store metadata about files and often contain valuable evidence even after files have been deleted.

Once the filesystem existed, the disk image still behaved like an ordinary file. To interact with it, we mounted the image using a loop device. Mounting tells Linux to treat the image exactly like a physical hard drive. After mounting, the filesystem became accessible through a directory, allowing us to create folders and files inside the virtual disk just as we would on a real storage device.

With the storage environment prepared, we began constructing a realistic employee workstation. We created directories such as HR, Projects, Personal, Downloads, Archive, and Temp. These folders were intentionally chosen because they resemble the layout commonly found on business computers. Inside them, we created realistic files including project documentation, meeting notes, payroll information, password files, activity logs, and personal files. The goal was not to create random data but to simulate normal user activity that investigators might later analyze.

The next phase represented the actions of the suspect. This was the most important part of the project because it generated the forensic artifacts that investigators would later examine. We intentionally renamed a project document to disguise its purpose, hid a confidential payroll file by changing its filename to begin with a period, deleted a personal photograph using the rm command, securely destroyed a password file using shred, and moved a USB activity log into another directory. Although these actions appear simple, each one leaves different traces inside the filesystem. Some files remain recoverable, some metadata changes, and some evidence becomes much harder to recover. Understanding these differences is one of the fundamental skills of a DFIR analyst.

After completing the evidence manipulation, we treated the image as completed forensic evidence. We safely unmounted the filesystem to ensure no further modifications would occur. We then calculated a SHA-256 cryptographic hash of the completed image. This hash acts as the digital fingerprint of the evidence. If even a single byte changes in the future, the hash value will also change. This is why forensic investigators always calculate and verify hashes to prove the integrity of digital evidence.

Finally, we documented the project by creating an answer key and a quality assurance checklist. The answer key records every intentional modification performed during dataset creation so that investigators or instructors can compare their findings against the known ground truth. The QA checklist confirms that every stage of the dataset was completed successfully and that the image is ready for forensic analysis.

The DFIR Perspective

As a Digital Forensics and Incident Response (DFIR) professional, you are not interested in the commands themselves. Your real objective is to understand the story they tell. Every user action—creating, renaming, moving, hiding, or deleting a file—leaves traces within the filesystem. Your job is to collect those traces, reconstruct the sequence of events, determine what happened, and explain your findings clearly and objectively.

This project taught you how a forensic dataset is built before any investigation begins. In real investigations, you will rarely create the evidence yourself. Instead, you will receive an already acquired forensic image. Because you now understand how that image is structured internally and how different user actions generate artifacts, you are better prepared to analyze evidence, recover deleted files, interpret filesystem metadata, and reconstruct user activity.

The Most Important Lesson

Digital forensics is not about memorizing Linux commands. It is about understanding how operating systems store information, how users interact with that information, how those interactions leave forensic artifacts, and how investigators use those artifacts to reconstruct the truth.

If you remember this one idea, you've understood the real purpose of the entire assignment.
